Summary of this project

This project is now deployed at yorubasystems.com. The orginal scope of this project was for a simple, online vocabulary quiz. This page is supposed to document the interesting parts of my research and the software development process.

Background

The system of writing the Yoruba language using latin-based characters was not the first, but for historical reasons, has become our de facto standard.

Unfortunately, the bus on which unicode code-points for latin characters were being standardised, left about 20 years ago.

Thankfully, some smart unicode people developed the 'combining diacritical marks' workaround (which I won't go into on this page). Being a workaround however, and not a core standard, it's use in today's 'web and information standards world' is not optimal.

The Challenge

The web is now the way most of the world consumes it's written information - so, not so much the printed media anymore. See the problem here? What happens when a language can't easily be plugged into the web?

So, my main 'interesting challenge' would be discovering for myself whether I could create a web application whose source data was from a character set for which some unicode code-points do not currently exists. Eh?

Research

In the Linux world, the bash interpreter will take an unix hex unicode code-point from stdin and give us back the corresponding unicode character in stdout:

Ctrl + Shift + u 0 0 e 9 # outputs: é echo -e '\u00e9' # (as an escape sequence) outputs: é

Developers need to be aware of which unicode hex formats are understood by the various interpreters that make up their web applications. Developers then need to use reliable pipelines to encode, decode, input, output, store and transmit Yoruba language character set data between these different interpreters.

Some of the default escape sequence formats I found were:

unix hex unicode refs used by bash interpreters:

\u00E9

xml unicode code-point used by most interpreters:

&#x00E9;

html cer used by browsers (which are interpreters):

&#233;

unix-like unicode used by php interpreter:

\u{00E9}

Then, there's also the font support for this workaround to consider. Monospaced and most specialist font families seem unable render even correctly decoded combining diacritical marks.

At this point in the research, my excitement levels had become 'quite high'.

I then had the idea that the simplest form of web application in which the reliable interpreting of these weird Yoruba characters could be demonstrated, would be a language quiz app. Still 'doing research', I then created a yoruba language quiz bash shell program as a kind of prototype of how such an application would handle JSON source data like this:

{ "englishPhrase": "'hat'", "yorubaPhrase": "'fil\\u{00E0}'", "category": "", "shortAudioFile": "" }

It worked as expected. From that point, I was confident enough to start developing a web application, briefly summarised as follows:

Application Architecture

Classic, monolithic, LAMP stack architecture would do for now. So, something like an Ubuntu Linux server operating system would be running an Apache http server, on which PHP and MySQL modules would need to be installed. All very tightly coupled, I know.

Database Design

Entity Relationship modelling used to create a Data Model. For an app with static data, so nothing changed or created by users, we could get away with reading the data from a JSON data structure. It would even perform much better. However, by this point I was beginning to develop 'big ideas' about the destiny of this web app. No point delaying. Only full CRUD functionality would do for where this application was going! A MySQL relational database, now!

Application Software Development

UML techniques used to model the quiz domain.

Implementation

The 'simple as possible' server-side web app implemented the Front Controller pattern, with one main page as the single entry point of control. The application was coded and database created within a local development environment. PHPUnit was installed locally using the Composer dependency manager, then used to conduct unit testing on our classes.

Deployment on AWS Cloud

A Single EC2 hypervisor instance was used to virtualise an Ubuntu Linux server OS, on which Apache web server was configured with various modules were built by hand (slow, but satisfying). So, web and database servers running on the same instance should be enough for now. Or so I thought. I won't go into that now.

Securing the Web App

Being a server-side web application gives an inherent layer of security. The Apache 'Web Application Firewall' module mod_security was also installed as an additional Application Layer inspector of web server traffic.

Monitoring the App

Web server logs, ubuntu server logs and AWS Cloudwatch.
